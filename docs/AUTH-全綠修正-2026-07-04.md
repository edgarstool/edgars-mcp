# 授權「全綠」修正（2026-07-04）

## 問題（為什麼有的客戶端一直掉紅）
在主網域 `mcp.edgars.tools/mcp` 上，舊版把兩種驗證方式設成**互斥**：
- `cloudflare_access_disable_builtin_oauth` 預設 **開**
- `cloudflare_access_allow_public_token_fallback` 預設 **關**

結果：主網域上唯一能進的方法是帶 `Cf-Access-Jwt-Assertion` 標頭（Cloudflare Access session）。
任何拿**正確 Bearer OAuth token** 的客戶端（Claude、ChatGPT 連接器、CIMD 客戶端）——
就算 token 完全有效——都會被 `_authenticate_public_request_via_cloudflare_access()` 直接擋成 401，
永遠走不到 bearer 檢查。這就是「授權不正統、有的綠有的紅」的根因。

## 修法（責任鏈，不管方法都全綠）
把 `_ensure_mcp_request_authorized()` 改成依序嘗試、任一通過即放行：
1. **Cloudflare Access JWT**：有帶且驗簽通過 → 綠。
2. **Bearer token**：靜態 `MCP_API_TOKEN` / 內建 OAuth 發出的 / CIMD 客戶端拿到的
   （皆由既有的 `bearer_token_is_authorized()` 判斷）→ 綠。
3. 兩者皆無有效憑證 → 回 401：
   - 受 CF Access 保護且完全沒帶 bearer → `cloudflare_access_required`（指引走 Managed OAuth）
   - 其餘 → 標準 MCP 401，`WWW-Authenticate` 指向 protected-resource-metadata（自動發現）

`HEALTH_PATH` 也改用同一條鏈，行為一致。

## 安全性
- **沒有放寬**：仍然「必須有一個有效憑證」才放行；無憑證一律 401。
- 只是不再把合法的 Bearer 憑證因為「主網域 + CF 模式」而誤擋。
- 未要求授權的設定（純本機/開發）維持原本短路放行行為。

## 影響檔案
- `server_http.py`：重寫 `_ensure_mcp_request_authorized()`；`HEALTH_PATH` 改用同鏈。
  （`_authenticate_bearer_request` / `_authenticate_public_request_via_cloudflare_access`
  變成未使用，保留不刪以縮小 diff。）

## 驗證
- `python -m py_compile server_http.py` 通過。
- 建議上線後實測：帶 CF Access session、帶 Bearer OAuth token、帶靜態 token 三種
  分別打 `POST /mcp` 都應 200；完全不帶 → 401 且 `WWW-Authenticate` 有 resource_metadata。
